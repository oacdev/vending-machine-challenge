🎰 Vending Machine — Interview Defense

PHP 8.3 · DDD + Hexagonal · Symfony Console · ~50 tests · @oacdev

60-Second Pitch

The one-liner: A fully in-memory vending machine simulator in PHP 8.3, built with Domain-Driven Design and Hexagonal Architecture — the domain layer has zero framework dependencies, and the only adapter is a Symfony Console REPL that proves the architecture is swappable.

Three bullets — memorise these

  • Architecture: Three layers (Domain / Application / Infrastructure), dependency direction enforced at CI by Deptrac. Domain knows nothing outside itself.
  • Money: Integer cents everywhere — no floats in arithmetic, zero rounding bugs by design. Floats only appear in display formatting.
  • State model: CoinBank and Inventory are immutable value objects; VendingMachine is the mutable aggregate root. Rollback on failure is free — just don't assign the new state.

Why Hexagonal?

  • Domain logic is pure PHP — no use Symfony\... anywhere inside src/Domain/.
  • Swap the CLI for HTTP: add one file src/Infrastructure/Http/VendingMachineController.php that calls VendingMachineService::handleInput(). Zero domain changes.
  • Swap for a GUI, gRPC, a test double — same story.
  • Deptrac enforces this at CI level — it will fail the build if anyone accidentally imports a Symfony class into Domain.

The Deptrac Rules (deptrac.yaml)

layers:
  Domain:          src/Domain/
  Application:     src/Application/
  Infrastructure:  src/Infrastructure/

ruleset:
  Domain:          []           # depends on nothing
  Application:     [Domain]
  Infrastructure:  [Application, Domain]

Adding a REST adapter — exactly what changes

  1. Add src/Infrastructure/Http/VendingMachineController.php — inject VendingMachineService, map HTTP verbs to handleInput(), translate VendingResult to JSON responses.
  2. Register it in your framework's routing config.
  3. Zero changes to Domain/, Application/, or any existing file.
State caveat: the current machine is instantiated fresh per CLI run (in-memory, no persistence). A REST adapter would need a shared machine instance per session/machine-id, which means adding a repository port and a persistence adapter too. That's the next architectural step.

Immutability: final readonly — PHP readonly means constructor-assigned properties cannot be reassigned. Every "mutation" copies the $coins array (PHP copy-on-write), modifies the copy, and returns new self($coins). The original is unchanged.

VendingResult / VendingResultType — Result pattern

enum VendingResultType { case Dispensed; case CoinsReturned; case Error; }

final readonly class VendingResult
{
    public VendingResultType $type;
    public ?Product         $product;
    public array            $change;   // Coin[]
    public string           $message;

    public static function dispensed(Product, array): self
    public static function coinsReturned(array): self
    public static function error(string): self
}
Why Result object, not exceptions? Exceptions model unexpected failures. At the service boundary, "insufficient funds" is an expected outcome — it should be a value, not exceptional flow. The domain does throw exceptions internally (DomainException subclasses), but the Application layer catches them and converts to VendingResult before returning to Infrastructure.

⚠ naming smell VendingResultType::Error is overloaded — it's returned for actual errors AND for info messages (coin-inserted feedback, SERVICE confirmation). A Message or Info case would be cleaner.

End-to-End Purchase Flow

Happy path: insert coins → buy product → receive change

1
CLI REPL reads user input via QuestionHelper::ask() in an infinite while(true) loop (VendingMachineCommand.php:50-71).
2
Insert coin: input "0.25" → handleInput("0.25") → parseCoin() returns Coin::Quarter → machine->insertCoin(Quarter) appends to $insertedCoins[] and adds 25 to $insertedAmountInCents. Returns VendingResult::error("Inserted 0.25. Current balance: 0.25.").
3
Select product: input "GET-SODA" → handleInput matches str_starts_with("GET-") → Product::tryFrom("soda") → machine->selectProduct(Soda).
4
Stock check: inventory->hasProduct(Soda) → count > 0 → OK. Else throw OutOfStockException.
5
Funds check: insertedAmountInCents (150) < price (150)? No → OK. Else throw InsufficientFundsException.
6
Overpayment: $overpayment = 150 - 150 = 0. Zero → exact change path.
7a
Exact change path: $this->coinBank = $this->coinBank->addCoins($this->insertedCoins). The inserted coins go into the bank. Done.
7b
Overpayment path (e.g. paid $1.00 for $0.65 water, overpayment = 35¢):
① $updatedBank = $this->coinBank->addCoins($this->insertedCoins) — merge inserted coins in.
② $change = $updatedBank->makeChange(35) — greedy: Quarter(25) + Dime(10) = [Quarter, Dime].
③ If $change === null → throw CannotMakeChangeException — nothing is assigned yet, so machine state is unchanged.
④ $this->coinBank = $updatedBank->removeCoins($change) — commit the bank update.
8
$this->inventory = $this->inventory->dispense($product) — new Inventory with count--.
9
$this->insertedCoins = []; $this->insertedAmountInCents = 0 — reset balance.
10
Return VendingResult::dispensed($product, $change). CLI prints product name + change coins.

The merge-before-change decision

Inserted coins are added to the bank before makeChange() because the customer's own coins might be needed to make change. Example: bank is empty, user inserts $1.00 and buys $0.65 water — the dollar must be in the bank for the greedy pass to pick a Quarter + Dime from it.

Rollback by non-assignment

$updatedBank is a local variable. If makeChange() returns null → CannotMakeChangeException is thrown → the function exits without ever writing to $this->coinBank. The aggregate's state is untouched. No try/finally, no explicit undo — just structural purity of immutable VOs.

CannotMakeChange → coin refund (service layer)

// VendingMachineService.php:58-68
try {
    return $this->machine->selectProduct($product);
} catch (OutOfStockException $e) {
    return VendingResult::error($e->getMessage());
} catch (InsufficientFundsException $e) {
    return VendingResult::error($e->getMessage());
} catch (CannotMakeChangeException) {
    $coins = $this->machine->returnCoins();   // refund!
    return VendingResult::coinsReturned($coins);
}

Note: CannotMakeChange does not propagate the exception message — the user just gets their coins back with no explanation of why the purchase failed. This is a UX gap.

Coin Selection Algorithm — Deep Dive

The full source

// CoinBank.php:90-120
/**
 * Greedy change algorithm: largest denomination first.
 *
 * @return array<Coin>|null  Coins for change, empty array for exact change, null if impossible
 */
public function makeChange(int $amount): ?array
{
    if ($amount === 0) {
        return [];              // ① zero → exact change, no coins needed
    }

    $available = $this->coins;  // ② PHP copy-on-write — function is PURE, no mutation
    $change    = [];
    $remaining = $amount;

    $denominations = [Coin::Dollar, Coin::Quarter, Coin::Dime, Coin::Nickel]; // ③ largest first — ORDER IS LOAD-BEARING

    foreach ($denominations as $coin) {
        while ($remaining >= $coin->value && $available[$coin->value] > 0) { // ④ bounded-supply guard
            $change[] = $coin;
            $available[$coin->value]--;
            $remaining -= $coin->value;
        }
    }

    if ($remaining !== 0) {
        return null;            // ⑤ cannot satisfy amount → null (NOT empty array)
    }

    return $change;
}

Step-by-step trace: buy Water ($0.65), pay $1.00

Pass Quarter  (25): 35 ≥ 25, available > 0 → take Quarter (rem = 10, available[25]--)
Pass Quarter  (25): 10 < 25 → stop

Pass Nickel    (5): 0 < 5   → stop

Change: $0.25, $0.10

Why greedy? The canonical-system argument

A coin system is canonical if the greedy algorithm (always take the largest coin ≤ remaining) produces an optimal solution (fewest coins) for every possible amount — assuming unlimited supply.

The US denomination set {1¢, 5¢, 10¢, 25¢, 50¢, 100¢} is canonical. Our subset {5¢, 10¢, 25¢, 100¢} is also canonical (amounts always multiples of 5, and greedy is optimal for all of them).

Proof sketch: You can verify canonicality by checking every amount from 1¢ to the second-largest denomination. For {5,10,25,100}: check 5–99¢. Greedy always produces the minimum-coin count for each. No counterexample exists. (Formal proof via the Pearson / Magazine canonicality tests.)

Non-canonical example (why we can't always use greedy)

Denominations: {1, 3, 4}
Make change for 6:

Greedy: 4 + 1 + 1 = 3 coins
Optimal: 3 + 3    = 2 coins  ← greedy FAILS here

With {1,3,4} you must use dynamic programming.

The empty array vs null distinction — critical

Return valueMeaningChecked by caller
[] (empty array)Exact change — no coins needed. Successful purchase.if ($change === null) → won't throw

The amount-zero short-circuit (line 97) returns [] directly, so the greedy loop never runs for exact payments. The caller also skips makeChange() entirely when $overpayment === 0 via the if/else at VendingMachine.php:59.

⚠ The Greedy + Bounded Inventory Bug

This is the most important algorithm caveat. Know it cold.

Greedy is optimal for canonical denominations with unlimited supply. But this machine tracks limited coin counts. When the bank is running low, greedy can commit to a large coin, strand the remainder, and return null — even though a valid combination of smaller coins exists.

Worked counterexample (reproducible)

Bank: { Quarter: 1, Dime: 3, Nickel: 0 }
Need to make change for: 30¢

Greedy pass:
  Dollar  (100): 30 < 100 → skip
  Quarter  (25): 30 ≥ 25, available[25] = 1 → take Quarter  (rem = 5)
  Dime     (10):  5 < 10  → skip
  Nickel    (5):  5 ≥  5, available[5]  = 0 → skip (none left!)

remaining = 5 ≠ 0 → return null  ←── WRONG! Refuses the sale.

But 3 × Dime = 30¢ is a VALID solution with the available coins!
Greedy chose the Quarter first and stranded 5¢ it couldn't fill.

When does this actually matter in practice?

  • Machine starts with 10 of each coin → plenty of nickels → greedy rarely fails early in a session.
  • After many transactions, quarters could run out and nickels be depleted → bug becomes triggerable.
  • The SERVICE command restocks to 10 of each → resets the problem.

The correct fix: bounded-supply DP

/**
 * Bounded coin-change via dynamic programming.
 * Finds minimum coins from limited inventory, or returns null if impossible.
 *
 * @param array<int,int> $available  cent-value → count
 * @return array<Coin>|null
 */
private function makeChangeDP(int $amount, array $available): ?array
{
    // dp[i] = minimum Coin[] to make i cents, null if impossible
    $dp = array_fill(0, $amount + 1, null);
    $dp[0] = [];

    foreach (Coin::cases() as $coin) {
        $count = $available[$coin->value] ?? 0;
        for ($k = 1; $k <= $count; $k++) {
            for ($i = $amount; $i >= $coin->value; $i--) {
                if ($dp[$i - $coin->value] !== null) {
                    $candidate = [...$dp[$i - $coin->value], $coin];
                    if ($dp[$i] === null || count($candidate) < count($dp[$i])) {
                        $dp[$i] = $candidate;
                    }
                }
            }
        }
    }

    return $dp[$amount];
}

Complexity: O(amount × Σcount_per_denomination) time, O(amount) space. For typical vending values (<200¢ change) this is trivially fast.

Algorithm complexity of current greedy

Value
TimeO(amount / min_denomination) in the worst case — bounded by 4 denominations and max ~$2 change → O(1) effectively
SpaceO(n) where n = number of coins in change — also tiny
Passes through denominationsAt most 4 outer iterations; inner while bounded by coin count

Pureness of makeChange

Despite being an instance method on a readonly class, makeChange is a pure function: it operates on $available = $this->coins which is a PHP copy-on-write value (an array assigned to a local variable). No side effects. Same input → same output. Safe to call multiple times or from tests without state concerns.

Interview Q&A Bank

Click any question to expand the model answer.

Why did you choose greedy over dynamic programming?▶

Greedy is correct and optimal for the canonical coin set {5, 10, 25, 100}. A coin system is canonical if greedy always produces the minimum-coin solution for every possible amount — and this set satisfies that property (all amounts are multiples of 5; 10 = 2×5, 25 can always be replaced by 2×10+1×5, etc.).

DP would be necessary for non-canonical sets like {1, 3, 4} — where greedy chooses 4 for amount 6 (leaves 1+1=3 coins) but optimal is 3+3=2 coins. Since we're not adding arbitrary denominations, greedy is the simpler, correct choice.

Known caveat: greedy + bounded inventory can still fail to find a valid solution that exists. If pressed, acknowledge this and describe the bounded-supply DP fix.

Prove that {5, 10, 25, 100} is a canonical system.▶

A canonical coin system has the property that greedy (take largest coin ≤ remaining) yields the fewest coins for any target amount.

For our set, all amounts dispensed are multiples of 5¢ (all denominations are multiples of 5, and the minimum denomination is 5¢). You can verify canonicality by exhaustive check of all amounts from 5¢ to 99¢ in steps of 5 — greedy achieves the optimum for all 19 values. (E.g. 30¢ → quarter + nickel = 2 coins; no better: 3 dimes = 3 coins). This check is the basis of the Pearson canonicality test.

Also note: 100 = 4×25 = 10×10 = 20×5, 25 = 2×10+1×5. No denomination is "wasteful" in a way that greedy would over-commit to.

Why are CoinBank and Inventory immutable but VendingMachine is mutable?▶

CoinBank and Inventory are value objects — they represent values without identity. Immutability makes them safe to share, cache, and use as method return values without defensive copying. State transitions are explicit: you always get back a new instance, never mutate in place.

VendingMachine is an aggregate root — it has identity (there is one machine) and it owns a transactional consistency boundary across multiple value objects. Its state evolves over time as part of its purpose. Making it immutable would mean the service layer would have to thread the machine value everywhere and replace it after every operation — unnecessary complexity. The aggregate root being mutable is standard DDD practice; the trick is that its internal VOs are immutable, giving you the rollback-by-non-assignment property.

Is selectProduct() atomic / thread-safe?▶

In this implementation — yes, trivially, because the runtime is single-threaded (Symfony Console, synchronous PHP-CLI, one request at a time). There is no concurrency.

If you were to move this to a web context with multiple concurrent users, selectProduct would NOT be atomic: there's a check-then-act race between hasProduct and dispense, and between makeChange and removeCoins. Two concurrent requests could both pass the stock check and both dispense the last item.

Fixes for a concurrent world: optimistic locking (version field on the aggregate + retry on conflict), pessimistic locking (mutex/advisory lock per machine), or a proper event-sourced / CQRS model where purchase commands are serialised through a queue per machine.

Why use a Result object instead of exceptions across the service boundary?▶

Exceptions model exceptional, unexpected failures — things outside the normal control flow. "Insufficient funds" is an expected business outcome; it's part of the normal use-case path. Modelling it as an exception you must catch is leaky — it forces every caller to know the domain's exception taxonomy.

A VendingResult with a typed VendingResultType is a first-class return value. The CLI adapter dispatches on the type with a match expression, which is exhaustive (PHP will error if a case is unhandled). It's type-safe, composable, and doesn't couple the Infrastructure layer to domain exception types.

The domain does still use exceptions internally because within the domain they model true constraint violations. But they don't cross the Application→Infrastructure boundary — the service layer catches and converts them.

Where's the float risk? How did you handle it?▶

Floats are IEEE 754 binary representations — they cannot exactly represent most decimal fractions. 0.10 + 0.20 === 0.30 is false in PHP (and most languages).

The solution: store all money as integer cents. Every value in the domain is an int: Coin::Quarter->value = 25, Product::Water->priceInCents() = 65, $insertedAmountInCents. Integer arithmetic is exact.

Floats appear only in display formatting: number_format($cents / 100, 2). The result is a string used only for output — it never feeds back into any calculation. The division is display-only. This is the correct boundary: floats for humans, ints for machines.

How would you test the CLI adapter? It has no tests.▶

Symfony Console ships a CommandTester utility that lets you simulate input/output without a real TTY. You'd wire it up like:

$app = new Application();
$app->add(new VendingMachineCommand());
$tester = new CommandTester($app->find('vending-machine'));
$tester->setInputs(['1.00', '0.25', '0.25', 'GET-SODA', 'EXIT']);
$tester->execute([]);
self::assertStringContainsString('Dispensed SODA', $tester->getDisplay());

The untested parts are the output formatting methods (displayDispensed, displayCoinsReturned, displayState, displayMenu). These are thin presentation logic — low risk — but the hexagonal claim is undercut by having zero adapter tests.

What happens to machine state when CannotMakeChangeException is thrown?▶

Nothing — state is left unchanged. Here's why:

In selectProduct, when overpayment > 0, the algorithm computes a local $updatedBank (by calling addCoins which returns a new instance). The critical thing is that $this->coinBank is NOT yet reassigned when makeChange is called. If makeChange returns null, we throw before the assignment on line 67 ($this->coinBank = $updatedBank->removeCoins($change)). So $this->coinBank still points to the original bank.

$this->inventory is also untouched (dispense happens on line 72, after the throw). $this->insertedCoins and $this->insertedAmountInCents are also untouched (reset on lines 73-74, after the throw).

The service then calls $machine->returnCoins(), which drains the (still-present) insertedCoins and returns them to the user. Perfect transactional semantics without a single try/finally or explicit undo.

Why do the domain exceptions extend DomainException?▶

PHP's SPL ships DomainException (extends LogicException) — it's the semantic bucket for "violations of domain invariants." Using it means a caller can catch the base DomainException to handle all business-rule failures generically if they want, or catch specific subtypes for fine-grained handling.

It also signals intent clearly: these exceptions are not RuntimeExceptions (unexpected infrastructure failures) or InvalidArgumentExceptions (bad input). They represent domain constraints violated during normal business logic.

All three are final — they can't be subclassed, preventing type hierarchy abuse.

Trade-offs of in-memory state. What happens on restart?▶

All state lives in the VendingMachine instance created at CLI startup. On restart: all inventory, coin counts, and balance are reset to the hardcoded starting values (10 coins each, 5 products each). There's no persistence layer.

Trade-offs:
Pros: zero infrastructure, instant start, trivial to test (just new VendingMachine(...)), no DB migration concerns.
Cons: no durability (power loss = all purchases forgotten), no multi-session state sharing, no history/audit log, can't run multiple machines, can't observe state from outside.

Adding persistence: define a VendingMachineRepository port in the Domain/Application layer, implement a JsonFileRepository or DatabaseRepository in Infrastructure. The aggregate needs a way to be reconstituted from stored state. This is the next architectural step — and it fits cleanly into the hexagonal structure.

How would you add a new coin denomination (e.g. 50¢)?▶

Currently requires changing 4 places — all in the domain:

  1. Add case HalfDollar = 50; to Coin enum.
  2. Add Coin::HalfDollar->value => 0 to CoinBank::empty().
  3. Add the same to CoinBank::withStock().
  4. Insert Coin::HalfDollar in the correct position (between Dollar and Quarter) in the $denominations array in makeChange().

The manual descending-order invariant in step 4 is the fragile part — if you insert it in the wrong position, greedy produces sub-optimal results.

Better design: derive $denominations from Coin::cases() sorted descending: usort($coins, fn($a,$b) => $b->value - $a->value). Then adding a denomination only requires editing the enum + the two initialisation arrays.

Also update parseCoin() in the service and the CLI display.

What would you refactor first?▶
  1. Fix the DP change algorithm — the greedy+bounded-inventory bug is a correctness issue that can cause sale rejections when valid change exists. Highest real-world impact.
  2. Fix InsufficientFundsException message — one-line fix, adds product name to the message.
  3. Sort denominations dynamically — remove the manual ascending-order invariant in makeChange; derive from enum cases sorted descending. Makes adding denominations safe.
  4. Split VendingResultType::Error — add an Info or Message case for non-error feedback so errors and info are distinguishable in the CLI adapter.
  5. Add CommandTester tests for the CLI adapter — closes the coverage gap on the only infrastructure adapter.
Why use Deptrac? Isn't it overkill for a small project?▶

In a small project, yes, it's belt-and-suspenders. In a real codebase: absolutely necessary. The hexagonal architecture claim is only as good as the enforcement. Without Deptrac, a junior developer accidentally adds use Symfony\Component\Console\... inside Domain and the architecture silently degrades. PHPStan won't catch it. Code review misses it. Deptrac fails the build instantly.

It also makes the architecture self-documenting: the deptrac.yaml file is a machine-readable spec of the intended dependency structure that can be audited in code review.

The cost is one YAML file and a dev dependency. The benefit is a continuous integration guarantee that the domain stays pure.

Which PHP 8.3 features does this project rely on?▶
  • Enums (8.1+) — Coin (int-backed) and Product (string-backed) with methods. Core to the domain model.
  • readonly classes (8.2+) — final readonly class CoinBank, Inventory, VendingResult. Immutability enforced by the language, not by convention.
  • Constructor property promotion (8.0+) — private function __construct(private array $coins).
  • Named arguments (8.0+) — new self(type: ..., product: ..., change: ..., message: ...) in VendingResult.
  • match expression (8.0+) — parseCoin(), priceInCents(), CLI dispatch.
  • str_starts_with (8.0+) — str_starts_with($input, 'GET-').
  • declare(strict_types=1) — every file. Prevents implicit type coercions.
How would you add a REST API without changing any domain code?▶

Create src/Infrastructure/Http/VendingMachineController.php. It would:

  1. Receive an HTTP request (e.g. POST /coin {"value":"0.25"}).
  2. Translate the HTTP payload to a string command (e.g. "0.25").
  3. Call $this->service->handleInput("0.25").
  4. Translate VendingResult to a JSON response.

Zero changes to Domain or Application. The service is already the correct abstraction.

Caveat: you'd also need to solve state persistence (the machine can't be instantiated fresh per HTTP request) — that requires a repository port and adapter. But the domain itself stays unchanged.

What does PHPStan level 8 mean and why use it?▶

PHPStan's levels go 0–9 (0 = basic, 9 = strictest). Level 8 requires explicit types everywhere, catches implicit mixed types, validates array shapes via PHPDoc (array<int,int>, array<Coin>), flags dead code and always-true/false conditions.

The domain code carries precise array-shape annotations specifically to satisfy PHPStan 8. This means type errors in the change algorithm (e.g. mixing coins and ints) are caught statically before tests run, let alone production.

It's worth it because PHP is dynamically typed — without strict static analysis, type bugs are runtime surprises. PHPStan at level 8 is as close as PHP gets to a compiled type-safe language.

What does SERVICE do exactly? Who should be allowed to call it?▶

SERVICE calls VendingMachine::service(CoinBank, Inventory) which replaces both VOs and resets inserted coins/amount to zero. The service layer hardcodes the restock values: 10 of each coin, 5 of each product.

Anyone who knows the command can call it from the CLI — there's no auth guard. In a real machine, this would be an operator-only action requiring a physical key or admin auth.

Improvements: authentication (PIN, role check), parameterisable restock quantities, audit log of service events.

Are there any consistency invariants that could be violated?▶

The aggregate maintains these invariants:

  • insertedAmountInCents == sum of all coin values in insertedCoins.
  • After a successful purchase: insertedAmountInCents == 0 and insertedCoins == [].
  • After a failed purchase (any exception): state is unchanged (no partial mutations).
  • CoinBank counts are always non-negative... in practice (no guard against negatives in removeCoins).

The first invariant is maintained because both are updated atomically in insertCoin(). The third is maintained by the immutable VO + rollback-by-non-assignment design.

The one real invariant risk: removeCoins() has no floor check, so a bug elsewhere could drive a coin count negative. Adding an assertion or invariant check there would be defensive programming.

Why no database? How would you add one?▶

No DB is intentional for this challenge scope — it's a CLI simulator, not a production service. State lives in memory for one session.

To add persistence in hexagonal architecture:

  1. Define VendingMachineRepositoryInterface in src/Domain/ (the port) — methods like findByMachineId(string): VendingMachine, save(VendingMachine): void.
  2. Inject it into VendingMachineService.
  3. Implement src/Infrastructure/Persistence/SqliteVendingMachineRepository (the adapter).
  4. Wire up via DI in the CLI bootstrap (bin/vending-machine).

The domain aggregate needs to be reconstitutable from stored state — either via a rich constructor or via an event-sourced approach (replay events to rebuild state).

Coin::fromAmount() is defined but seemingly unused in production. Why?▶

Good catch. Coin::fromAmount(int $cents) was likely written before or alongside parseCoin() in the service. parseCoin() operates on a string ("0.25") and uses a match expression. fromAmount() takes an int (cents) and delegates to Coin::from($cents) which is the native backed-enum from-value lookup.

They serve different inputs. fromAmount would be useful if the code ever received an int amount (e.g. from an API or DB). The CLI only ever receives string input.

Options: keep it as a useful utility (it IS tested), remove it to reduce surface area, or use it in the service layer to DRY up parseCoin() by converting string→float→int→Coin (though a match is cleaner and avoids float intermediaries).

How confident are you in the test coverage?--

Strong for domain logic (~80-90% branch coverage by inspection). The CoinBank and VendingMachine tests are thorough — they explicitly test rollback, immutability, and the key boundary conditions in makeChange.

Gaps: CLI adapter (zero tests), some VendingResult factory paths only exercised indirectly, and critically — no test for the greedy+bounded-inventory bug (the "impossible null when valid combo exists" case is not covered).

To increase confidence: add CommandTester tests for the CLI, add a mutation testing run with Infection (it mutates operators/conditionals and verifies tests catch the mutation), and add the bounded-inventory counterexample as a test.

Taking It Further — Expansion Scenarios

1. REST/HTTP Adapter

Effort: 1 new file + framework routing. Zero domain changes.
  • Add src/Infrastructure/Http/VendingMachineController.php that injects VendingMachineService.
  • Map HTTP verbs: POST /coins, POST /products/{id}, DELETE /coins (return), POST /service.
  • Translate VendingResult to JSON: {"type":"dispensed","product":"soda","change":["0.25","0.10"]}.
  • New problems: stateless HTTP → machine state must be persisted between requests (see persistence below). Concurrency (multiple simultaneous requests → race conditions on stock/balance). Session management (which machine instance does this user belong to?).

2. Persistence Layer

Effort: 1 domain port interface + 1 infrastructure adapter. Moderate.
  • Define VendingMachineRepositoryInterface in Domain: findById(string): VendingMachine, save(VendingMachine): void.
  • Implement JsonFileMachineRepository or DoctrineMachineRepository in Infrastructure.
  • Options: serialize the aggregate state to JSON on every mutation; or event-sourced (store events like CoinInserted, ProductDispensed and replay to rebuild state).
  • Event sourcing gives you a full audit log and time-travel debugging for free — great fit for financial/vending contexts.

3. Fix the Change Algorithm (Bounded-Supply DP)

Effort: Replace one method in CoinBank. High correctness value.
  • Replace greedy with bounded-coin DP (see algorithm section for sketch).
  • O(amount × Σcounts) — trivially fast for vending amounts (<200¢).
  • Add the greedy counterexample as a regression test first.
  • Consider keeping greedy as the fast path and falling back to DP only when greedy returns null.

4. Configurable Denominations & Products

Effort: Medium. Affects Domain + Infrastructure config.
  • Move coin denominations out of the Coin enum into a config file / runtime input.
  • Detect non-canonical denomination sets at boot and switch to DP automatically.
  • Move product catalogue (names + prices) to config/DB — operators can update prices without a deployment.
  • Requires rethinking parseCoin() (currently a match on hardcoded strings) to dynamically map input strings to denomination values.

5. Concurrency & Multi-Machine

Effort: Significant. Architectural shift.
  • Add a machine ID. Multiple physical machines share a DB but have independent state.
  • Optimistic locking: add a version field to the aggregate. On save, check version hasn't changed; retry on conflict.
  • Or pessimistic locking: acquire a DB advisory lock per machine-id for the duration of a transaction.
  • CQRS/Event Sourcing: commands go through a serialised command handler per machine-id; events are the source of truth.

6. Low-Stock & Low-Coin Alerting

Effort: Small. Domain events + observer.
  • Emit domain events: ProductLowStockEvent, CoinDenominationLowEvent.
  • An observer (in Application or Infrastructure) subscribes and sends alerts (email, webhook, dashboard metric).
  • Thresholds configurable: alert when any denomination drops below N coins.

7. Multi-Currency / i18n

Effort: Medium. Replaces Coin enum model.
  • Introduce a Money value object (amount + currency code + minor-unit exponent).
  • Coin denominations become data, not enum cases — loaded from a currency config.
  • Change algorithm needs to be aware of the minimum representable unit.
  • Prices and change formatted according to locale.

8. Telemetry & Observability

Effort: Small (structured logging) to Large (full APM).
  • Structured JSON logging on every transaction: product, amount, change, timestamp, machine-id.
  • Metrics: total revenue, dispensed-per-product, failed-change rate, return rate — feed to Prometheus/Grafana.
  • Tracing: distributed trace per user session if multiple services are involved.

9. Hardware Integration

Effort: Large. New port/adapter pair.
  • The domain models coin insertion as a method call. A hardware coin acceptor would be an adapter implementing a CoinAcceptorInterface port — it listens for pulses from the coin mechanism and calls machine->insertCoin().
  • Dispensing motors, bill acceptors, refund mechanisms all map to additional ports.
  • This is exactly what hexagonal architecture is designed for: the domain doesn't care whether insertCoin came from a CLI string, an HTTP request, or a hardware pulse.

10. CI/CD Pipeline

Effort: Small. Configuration only.
  • Add .github/workflows/ci.yml: on push/PR, run composer qa (cs-check, phpstan, deptrac, phpunit).
  • Add coverage reporting: phpunit --coverage-clover + upload to Codecov/Coveralls.
  • Add mutation testing: Infection PHP — mutates source operators and verifies tests catch each mutation. Surfaces under-tested logic that line coverage misses.
  • Matrix test against PHP 8.3 + 8.4 to stay ahead of deprecations.

11. Admin / Operator UI

Effort: Medium.
  • Separate operator interface (CLI or web) with auth.
  • View machine state: inventory, coin counts, revenue.
  • Restock with custom quantities (vs the hardcoded SERVICE=10).
  • Transaction history / receipts.

What Went Wrong — Holded Feedback, Dissected

This is the rejection feedback from Holded verbatim, broken down into root causes. The goal is not to defend the choices but to understand precisely why each one missed the bar for a Senior Backend Developer role.

1. Application layer coupled to Infrastructure — the biggest miss

Feedback: "VendingMachineService depende directamente del input que viene de la UI (parseo de prompts) haciendo que el servicio no sea válido para cualquier otra entrada de datos."

What the code does: VendingMachineService::handleInput(string $input) receives a raw CLI string ("0.25", "GET-SODA"), parses it with parseCoin() and string operations, and routes it internally. The service itself is the parser and the router.

// VendingMachineService.php — the problem
public function handleInput(string $input): VendingResult
{
    $input = strtoupper(trim($input)); // ← CLI-specific normalisation

    if (str_starts_with($input, 'GET-')) { // ← CLI command format hardcoded here
        $productName = strtolower(substr($input, 4));
        $product = Product::tryFrom($productName);
        // ...
    }
    $coin = $this->parseCoin($input); // ← maps "0.25" → Coin::Quarter
}

private function parseCoin(string $input): ?Coin
{
    return match ($input) {
        '0.05' => Coin::Nickel,  // ← CLI decimal format baked into Application layer
        '0.10' => Coin::Dime,
        '0.25' => Coin::Quarter,
        '1.00' => Coin::Dollar,
        default => null,
    };
}

Why this is wrong: The Application layer now "knows" that coins are typed as decimal strings like "0.25" and products are prefixed with GET-. A REST API might send {"coin": 25} (cents int) or {"product": "soda"}. A different CLI might use INSERT:25 or BUY:SODA. Each adapter would need to format its input to match the service's internal parsing expectations — the exact opposite of what hexagonal architecture promises.

The correct shape: The service should receive already-parsed, typed domain objects. The CLI adapter translates "0.25" → Coin::Quarter itself, then passes the Coin to the use case. The service never sees raw strings.

2. No use cases + no CQRS

Feedback: "no hay casos de uso diferenciados en la capa de aplicación; tampoco hay separación entre procesos de lectura/escritura (CQRS)"

What the code does: All operations collapse into one method: handleInput(string). There is no InsertCoinUseCase, no SelectProductUseCase, no ReturnCoinsUseCase, no ServiceMachineUseCase. They are if/elseif branches inside a single class.

Why this matters:

  • Adding a new operation (e.g. GET-STATE) means modifying VendingMachineService — this is an OCP violation (see #4).
  • CQRS means commands (mutate state, return nothing or a minimal ack) are structurally separated from queries (read state, no side effects). handleInput is both at once — it reads state to build the balance message after inserting a coin, and it writes state. A GetMachineStateQuery with its own handler would be the correct query side; command handlers would own the write side.
  • Without distinct use cases you cannot independently test, version, or authorise individual operations.

3. No persistence layer

Feedback: "no hay una capa de persistencia definida. El estado de la máquina es volátil y permanece solo mientras la instancia de VendingMachine permanezca viva."

What's missing: Not just the implementation — the port. Hexagonal architecture requires a VendingMachineRepositoryInterface defined in the Domain or Application layer so the aggregate is persistence-ignorant but the architecture acknowledges that persistence exists. The absence of the interface means the architecture has no socket for an adapter to plug into.

For a coding challenge, in-memory is a reasonable implementation choice. But the port (interface) should still be defined and the CLI bootstrap should use an InMemoryVendingMachineRepository that implements it. That's the difference between "I chose in-memory for simplicity" and "I didn't think about persistence."

4. SOLID violations: SRP and OCP in VendingMachineService

Feedback: "VendingMachineService incumple el single responsibility principle o el open-close principle. En cambio, sí se hace buen uso de la inversión de dependencias."

SRP violation — count the reasons to change VendingMachineService:

  1. The CLI string format for coins changes (e.g. from "0.25" to "25c") → change the service.
  2. A new product is added → potentially change the service routing.
  3. A new command is added (e.g. GET-STATE) → change the service.
  4. Error message wording changes → change the service.
  5. The SERVICE restock values change → change the service.
  6. The balance display format changes → change the service.

Each of those is a separate reason to change one class. SRP says one class should have one reason to change.

OCP violation: Adding any new command requires opening VendingMachineService and adding a new branch. A Command + CommandBus pattern would let you register new handlers without ever touching the bus or the service orchestrator.

What was done right: VendingMachine is injected into VendingMachineService — not created inside it. That is correct DI / Dependency Inversion. The feedback acknowledges this explicitly.

5. Missing design patterns: Repository and Command

Feedback: "no se aprecia el uso de patrones de diseño claros. Por poner dos ejemplos, el ya nombrado Repository o el patrón Command serían evidentes en este tipo de prueba."

Repository: Not just "missing persistence" — the Repository pattern defines a collection-like interface for retrieving and saving aggregates, decoupling the domain from the storage mechanism. It should have been defined even if backed by an in-memory array.

Command pattern: Two forms are relevant here:

  • Command object: Each user intention becomes a typed DTO — InsertCoinCommand(Coin $coin), SelectProductCommand(Product $product). The CLI translates raw input into Command objects before any Application-layer code runs.
  • Command Bus: A dispatcher that maps each Command type to its handler. Adding a new operation = add a new Command + Handler pair, register it. No existing code touched.

For a senior-level challenge, both patterns are table stakes. Their absence signals unfamiliarity with enterprise application architecture.

6. No domain events

Feedback: "el dominio está bastante bien modelado y VendingMachine funciona bien como aggregate root. Se podría haber ido un poco más allá e implementar eventos de dominio."

This is the feedback item with the most positive framing — the domain modeling is praised. The gap is domain events.

What's missing: When selectProduct() succeeds, it returns a VendingResult. That's synchronous, tightly coupled output. The richer pattern is for the aggregate to record domain events internally during the operation:

  • ProductDispensedEvent($product, $change, $timestamp)
  • CoinsInsertedEvent($coin, $newBalance)
  • CoinsReturnedEvent($coins)
  • MachineServicedEvent($operator, $timestamp)

After the aggregate operation completes, the Application layer calls $machine->releaseEvents(), gets the events, and dispatches them to an event bus. Listeners handle side effects (persist to event store, send metrics, trigger notifications) without polluting the aggregate.

Domain events are also the foundation for event sourcing — replay events to reconstruct any past state of the machine.

7. VendingResult belongs in Application, not Domain

Feedback: "sería discutible si estos DTOs pertenecen a la capa de dominio o si estarían mejor ubicados en la de aplicación."

VendingResult and VendingResultType live in src/Domain/. But they model the output of a use case — a concept that belongs to the Application layer. The Domain shouldn't know about "how results are packaged for callers." The domain aggregate should return a plain value or record an event; the Application use case wraps that in a response DTO.

Conversely, keeping them in Domain means the Domain is "aware" of the fact that there are different caller scenarios (Dispensed vs CoinsReturned vs Error) — that's orchestration logic, not domain logic.

Taking it From Here — Concrete Fixes

These are the exact code-level changes that would address the Holded feedback. Each fix is independent — they can be applied incrementally. Start with #1 and #2, they unlock everything else.

Fix 1 — Decouple parsing from the service: Command objects

Move all input translation into the Infrastructure layer. The Application service receives typed domain objects, never raw strings.

// src/Application/Command/ — Command DTOs (plain data carriers, no logic)
final readonly class InsertCoinCommand
{
    public function __construct(public readonly Coin $coin) {}
}

final readonly class SelectProductCommand
{
    public function __construct(public readonly Product $product) {}
}

final readonly class ReturnCoinsCommand {}
final readonly class ServiceMachineCommand {}


// src/Infrastructure/Cli/VendingMachineCommand.php — translates CLI strings to Commands
private function parseInput(string $raw): object|null
{
    $input = strtoupper(trim($raw));

    if ($input === 'RETURN-COIN') return new ReturnCoinsCommand();
    if ($input === 'SERVICE')     return new ServiceMachineCommand();

    if (str_starts_with($input, 'GET-')) {
        $product = Product::tryFrom(strtolower(substr($input, 4)));
        return $product ? new SelectProductCommand($product) : null;
    }

    $coin = match ($input) { // ← CLI format knowledge stays in CLI layer
        '0.05' => Coin::Nickel,
        '0.10' => Coin::Dime,
        '0.25' => Coin::Quarter,
        '1.00' => Coin::Dollar,
        default => null,
    };
    return $coin ? new InsertCoinCommand($coin) : null;
}

A REST controller would do the same translation from JSON body to Command objects — no changes to Application layer needed.

Fix 2 — Separate use cases, one class per operation

// src/Application/UseCase/InsertCoin.php
final class InsertCoin
{
    public function __construct(private readonly VendingMachineRepositoryInterface $repository) {}

    public function __invoke(InsertCoinCommand $command): InsertCoinResponse
    {
        $machine = $this->repository->get();
        $machine->insertCoin($command->coin);
        $this->repository->save($machine);

        return new InsertCoinResponse(
            coin: $command->coin,
            newBalanceInCents: $machine->getInsertedAmount(),
        );
    }
}

// src/Application/UseCase/SelectProduct.php
final class SelectProduct
{
    public function __construct(private readonly VendingMachineRepositoryInterface $repository) {}

    public function __invoke(SelectProductCommand $command): SelectProductResponse
    {
        $machine = $this->repository->get();
        $result  = $machine->selectProduct($command->product);
        $this->repository->save($machine);

        return SelectProductResponse::fromDomainResult($result);
    }
}

Each use case has one responsibility, one constructor dependency, one reason to change. Adding GetMachineState or SetProductPrice means adding a new file — nothing existing is modified.

// src/Infrastructure/Persistence/DoctrineVendingMachineRepository.php — production step

Fix 4 — CQRS: separate read model from write model

// Write side — commands mutate state, return minimal ack
interface CommandBusInterface
{
    public function dispatch(object $command): mixed;
}

// Read side — queries never mutate, return DTOs
final readonly class GetMachineStateQuery {}

final class GetMachineStateHandler
{
    public function __construct(private readonly VendingMachineRepositoryInterface $repository) {}

    public function __invoke(GetMachineStateQuery $q): MachineStateDTO
    {
        $machine = $this->repository->get();
        return new MachineStateDTO(
            balanceInCents: $machine->getInsertedAmount(),
            inventory: $machine->getState()['inventory']->toArray(),
            coinBank: $machine->getState()['coinBank']->toArray(),
        );
    }
}

// CLI displayState() calls this query handler, not machine->getState() directly

This also fixes the display-state coupling: the CLI's displayState() currently reaches into VendingMachineService::getMachine()->getState() — a deep reach through layers. With CQRS, it calls the query handler instead.

Fix 5 — Domain events in the aggregate

// src/Domain/Event/DomainEvent.php — base marker interface
interface DomainEvent {}

// src/Domain/Event/ProductDispensedEvent.php
final readonly class ProductDispensedEvent implements DomainEvent
{
    public function __construct(
        public readonly Product $product,
        public readonly array   $change,    // Coin[]
    ) {}
}

// src/Domain/VendingMachine.php — aggregate records events
final class VendingMachine
{
    /** @var array<DomainEvent> */
    private array $domainEvents = [];

    public function selectProduct(Product $product): void  // no longer returns VendingResult
    {
        // ... existing logic ...
        $this->domainEvents[] = new ProductDispensedEvent($product, $change);
    }

    /** @return array<DomainEvent> */
    public function releaseEvents(): array
    {
        $events = $this->domainEvents;
        $this->domainEvents = [];
        return $events;
    }
}

// Application use case publishes events after saving the aggregate
public function __invoke(SelectProductCommand $cmd): void
{
    $machine = $this->repository->get();
    $machine->selectProduct($cmd->product);
    $this->repository->save($machine);

    foreach ($machine->releaseEvents() as $event) {
        $this->eventBus->publish($event);  // decoupled side effects
    }
}

Verdict: what was actually good

The feedback explicitly praises these — internalise them as strengths:
  • Domain modeling: VendingMachine as aggregate root is correct and well-implemented.
  • Value objects: CoinBank and Inventory as immutable VOs are well-done, functioning as domain model safeguards.
  • Dependency inversion: VendingMachine is injected, not created inside the service. DI is applied correctly.
  • DTOs for responses: Using VendingResult as a structured response (not raw arrays or primitives) is the right instinct — just wrong layer.
  • Unit test strategy: Testing important and sensitive flows is adequate. The rollback test is particularly solid.

The core lesson

The domain layer is solid. The gap is the Application layer — it was treated as a thin dispatcher when it should be the architectural heart of the system: typed use cases, typed commands, typed responses, CQRS separation, and a repository port that makes persistence pluggable. A senior backend role in a DDD shop expects to see all of these as defaults, not extras.
